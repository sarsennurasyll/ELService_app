# Пользовательские сценарии

## Основной бизнес-сценарий

Полный целевой сценарий ELService:

```text
Клиент
  -> создаёт заказ
  -> мастера получают заказ
  -> мастера отправляют предложения
  -> клиент выбирает мастера
  -> предложение принимается
  -> заказ получает назначенного мастера
  -> создаётся чат
  -> клиент и мастер общаются
  -> мастер выполняет ремонт
  -> клиент оплачивает
  -> клиент оставляет отзыв
```

## Что реализовано в текущем проекте

Реализована техническая основа для середины сценария:

- регистрация/вход;
- категории;
- создание заказа;
- получение заказов;
- детали заказа;
- отправка предложения мастером;
- просмотр предложений клиентом;
- принятие предложения;
- назначение мастера;
- создание чата;
- HTTP API сообщений.

Не реализованы:

- реальные уведомления мастерам;
- автоматический подбор мастеров;
- оплата;
- отзывы через API;
- завершение ремонта через полноценный бизнес-процесс;
- push/WebSocket.

## Customer workflow

### 1. Первый запуск

Пользователь открывает приложение.

Маршрут:

```text
/
```

Экран:

```text
SplashPage
```

Дальше пользователь переходит на Login.

### 2. Регистрация

Маршрут:

```text
/register
```

Flutter отправляет:

```text
POST /api/v1/auth/register
```

После успеха:

- сохраняется access token;
- сохраняется refresh token;
- сохраняется session;
- пользователь переходит на `/home`.

### 3. Вход

Маршрут:

```text
/login
```

Flutter отправляет:

```text
POST /api/v1/auth/login
```

После успеха:

- сохраняется access token;
- сохраняется refresh token;
- сохраняется session;
- пользователь переходит на `/home`.

### 4. Главный экран клиента

Маршрут:

```text
/home
```

Экран:

```text
HomePage
```

Данные:

- категории загружаются через `GET /api/v1/categories`;
- список мастеров пока локальный mock.

Пользователь может нажать категорию или emergency banner и перейти к созданию заказа.

### 5. Создание заказа

Маршрут:

```text
/order/new
```

Экран:

```text
CreateOrderPage
```

Данные:

- категории загружаются через API;
- customer id берётся из сохранённой session.

После нажатия `Submit request` Flutter отправляет:

```text
POST /api/v1/orders
```

После успеха:

- увеличивается `ordersRefreshNotifier`;
- пользователь переходит на `/orders`.

### 6. Список заказов

Маршрут:

```text
/orders
```

Экран:

```text
OrdersPage
```

Flutter отправляет:

```text
GET /api/v1/orders
```

UI показывает:

- loading;
- error;
- empty;
- active/past orders.

Пользователь может открыть детали заказа.

### 7. Детали заказа

Маршрут:

```text
/order/details/:id
```

Экран:

```text
OrderDetailsPage
```

Flutter отправляет:

```text
GET /api/v1/orders/:id
```

UI показывает:

- статус;
- описание;
- адрес;
- дату;
- стоимость или estimate.

Кнопки `Call`, `Chat`, `Complete`, `Rate technician` пока не подключены к полноценной бизнес-логике.

### 8. Предложения по заказу

Маршрут:

```text
/order/:id/offers
```

Экран:

```text
OffersPage
```

Flutter отправляет:

```text
GET /api/v1/orders/:id/offers
```

Пользователь может принять предложение:

```text
PATCH /api/v1/offers/:id/accept
```

После принятия backend:

- ставит выбранному предложению `ACCEPTED`;
- ставит остальным предложениям `INACTIVE`;
- назначает мастера в заказ;
- меняет статус заказа на `ACCEPTED`;
- создаёт чат.

## Technician workflow

### 1. Вход мастера

Backend поддерживает роль:

```text
TECHNICIAN
```

Flutter auth может отправлять role при регистрации.

Полное переключение интерфейса по роли пока не реализовано.

### 2. Навигация мастера

Маршруты:

- `/tech`;
- `/tech/orders`;
- `/tech/calendar`;
- `/tech/earnings`;
- `/tech/profile`.

Страницы существуют как UI.

### 3. Отправка предложения

Маршрут Flutter:

```text
/tech/order/:id/offer
```

Экран:

```text
SendOfferPage
```

Flutter отправляет:

```text
POST /api/v1/offers
```

Backend проверяет:

- JWT;
- роль `TECHNICIAN`;
- заказ существует;
- заказ в статусе `PENDING`.

## Chat workflow

### 1. Создание чата

Чат создаётся после принятия предложения.

Также есть endpoint:

```text
POST /api/v1/chats
```

Он создаёт чат по `orderId` или возвращает существующий.

### 2. Список чатов

Backend endpoint:

```text
GET /api/v1/chats
```

Возвращает чаты текущего пользователя.

Flutter data/domain слой реализован.

Ограничение:

- текущий `ChatPage` ещё показывает mock-список и не использует repository.

### 3. Сообщения

Backend endpoints:

```text
GET /api/v1/chats/:id/messages
POST /api/v1/chats/:id/messages
```

Flutter `ChatMessagesPage`:

- загружает сообщения;
- отправляет сообщение;
- обновляет сообщения polling-таймером каждые 10 секунд.

Ограничение:

- маршрут на `ChatMessagesPage` не подключён в `AppRouter`.

## Admin workflow

Маршруты:

- `/admin`;
- `/admin/orders`;
- `/admin/technicians`;
- `/admin/analytics`;
- `/admin/settings`.

Flutter-страницы существуют.

Backend admin API не реализован.

## Сценарии, которые пока не закрыты

### Уведомления мастерам

Целевой сценарий требует, чтобы подходящие мастера получали уведомления.

Фактически:

- модель `Notification` есть в Prisma;
- backend API уведомлений нет;
- push-уведомлений нет;
- алгоритма подбора мастеров нет.

### Ремонт

Целевой сценарий включает выполнение ремонта.

Фактически:

- статусы заказов есть;
- отдельного workflow ремонта нет;
- изменения статуса через Flutter не реализованы.

### Оплата

Целевой сценарий включает оплату.

Фактически:

- API оплаты нет;
- Flutter экран оплаты не реализован;
- в профиле есть только UI-пункт `Payment methods`.

### Отзывы

Целевой сценарий включает отзыв.

Фактически:

- модель `Review` есть;
- API отзывов нет;
- UI кнопка `Rate technician` есть, но не подключена.

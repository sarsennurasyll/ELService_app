# Правила работы с ELService

Этот документ обязателен для всех разработчиков и ИИ-помощников, работающих с репозиторием.

## Общие правила разработки

- Перед началом работы прочитать этот файл и учитывать правила проекта.
- Выполнять только поставленную задачу и не переходить к следующему этапу без нового задания.
- Не писать лишний код, временные решения и код «на будущее».
- Не добавлять зависимости без явного задания.
- Перед изменением архитектуры необходимо получить подтверждение.
- Не менять архитектуру самостоятельно.
- Перед изменением существующего кода изучить связанные файлы и не затрагивать несвязанные части проекта.
- Любой новый файл должен быть самодостаточным и компилируемым на текущем этапе. Не создавать код, который не собирается в ожидании следующих задач. Если временное решение неизбежно, оно должно быть минимальным и явно отмечено как временное.

## Архитектура проекта

Проект использует Feature First и Clean Architecture.

Каждая функциональность располагается в `lib/features/<feature_name>` и развивается независимо от других функций. Внутри feature слои разделяются по ответственности:

- `data` — источники данных, DTO, мапперы и реализации репозиториев;
- `domain` — сущности, контракты репозиториев и сценарии использования;
- `presentation` — состояние, провайдеры и UI.

Слой `domain` не зависит от Flutter, Dio, JSON и деталей хранения данных. `presentation` не обращается к API напрямую. Зависимости направлены от внешних слоёв к внутренним.

## Организация папок

- `lib/app` — запуск приложения, маршрутизация, тема и конфигурация приложения.
- `lib/core` — общая техническая инфраструктура, не связанная с конкретной feature.
- `lib/shared` — общие элементы предметной области и переиспользуемые виджеты.
- `lib/features` — функциональные модули приложения.
- `lib/l10n` — локализация.

Не размещать код feature в `core` или `shared`. Общий код выносить туда только после подтверждённой необходимости использовать его в нескольких feature.

## Именование

- Названия файлов, папок, классов, функций и переменных писать на английском языке.
- Файлы и папки именовать в `snake_case`.
- Классы, перечисления и типы именовать в `PascalCase`.
- Функции, методы, переменные и параметры именовать в `camelCase`.
- Булевы значения начинать с `is`, `has`, `can` или `should`, когда это делает смысл понятнее.
- Имена должны описывать назначение и не содержать неясных сокращений.

## Комментарии

- Комментарии, README и документацию писать только на русском языке.
- Комментарии добавлять только там, где без них сложно понять причину решения или важное ограничение.
- Писать кратко и естественно, как опытный разработчик.
- Не использовать шаблонные фразы: «Данный метод», «Этот код предназначен», «Следующий блок», «Выполняем инициализацию», «Важно отметить».
- Не комментировать очевидный код.

## README

- README писать только на русском языке.
- Описывать назначение проекта, стек, структуру, запуск и важные особенности работы.
- Писать кратко, ясно и по существу.
- Поддерживать README в актуальном состоянии при изменении информации, которую он описывает.

## Riverpod

- Для управления состоянием использовать Riverpod.
- Провайдеры размещать рядом с feature в слое `presentation`, если они не относятся к инфраструктуре приложения.
- Глобальные инфраструктурные провайдеры размещать в `lib/core/providers`.
- Не использовать провайдеры как контейнер для несвязанной логики.
- Состояние и зависимости должны иметь понятные границы и минимальную область видимости.

## Dio

- Для HTTP-запросов использовать Dio.
- Настройку клиента, интерцепторы и общую обработку сетевых ошибок хранить в `lib/core/network`.
- Не создавать экземпляры Dio внутри виджетов, сценариев использования или репозиториев feature.
- API-вызовы размещать в источниках данных слоя `data`.
- Не передавать сетевые исключения в UI без преобразования в ошибки доменного уровня.

## Freezed и Json Serializable

- Freezed использовать для неизменяемых моделей состояния и моделей данных, когда нужны генерация `copyWith`, сравнение значений или объединения типов.
- Json Serializable использовать для DTO и преобразования JSON.
- Аннотации и сгенерированные части хранить рядом с исходной моделью.
- Не редактировать сгенерированные файлы вручную.
- Не смешивать DTO с доменными сущностями: преобразование выполнять через мапперы слоя `data`.

## UI

- Использовать Material 3.
- Экран, его состояние и локальные виджеты размещать в `presentation` соответствующей feature.
- Не размещать бизнес-логику, HTTP-запросы и преобразование DTO в виджетах.
- Предпочитать небольшие композиционные виджеты вместо больших файлов и монолитных экранов.
- Общие виджеты добавлять в `shared/widgets` только если они используются несколькими feature.
- Не создавать UI без отдельного задания.

## Принципы разработки

- SOLID: у класса одна чёткая ответственность, зависимости строятся через абстракции.
- DRY: не дублировать код; обобщать только при реальном повторном использовании.
- KISS: выбирать самое простое решение, которое закрывает текущую задачу.
- Clean Code: код должен быть читаемым, предсказуемым, компактным и легко тестируемым.

## Создание новой feature

- Перед созданием feature убедиться, что она соответствует согласованной предметной области и задаче.
- Создавать feature в `lib/features` с понятным именем в `snake_case`.
- Добавлять слои `data`, `domain` и `presentation` только когда они требуются текущей задачей.
- Не создавать пустые классы, сценарии использования, репозитории или экраны заранее.
- Подключение маршрутов, зависимостей и локализации выполнять только в рамках поставленной задачи.

## Git

- Не включать в коммит несвязанные изменения.
- Перед коммитом проверить список изменённых файлов и убедиться, что в нём нет сгенерированных или временных артефактов, не требуемых задачей.
- Один коммит должен отражать одну логически завершённую задачу.
- Не использовать принудительные операции, которые могут удалить чужие изменения, без явного подтверждения.

## Conventional Commits

Сообщения коммитов оформлять в формате `type: описание`, писать на русском языке и в прошедшем времени.

- `feat` — новая функциональность.
- `fix` — исправление ошибки.
- `refactor` — изменение структуры кода без изменения поведения.
- `docs` — документация.
- `test` — тесты.
- `style` — форматирование и изменения стиля кода.
- `build` — зависимости и настройки сборки.
- `chore` — служебные изменения.

Примеры:

- `feat: добавил экран авторизации`
- `feat: реализовал создание заказа`
- `fix: исправил обработку ошибок авторизации`
- `refactor: вынес общие виджеты`
- `docs: обновил README`
- `test: добавил тесты авторизации`
- `style: отформатировал код`
- `build: обновил зависимости`
- `chore: создал структуру проекта`
- `chore: настроил статический анализ`
